## 项目核心思想

### 交流效率最大化

在软件开发过程中，人与大模型之间的交流存在显著的效率瓶颈：
- **传统打字交流**：速度约为 1-2 比特/秒，信息传递缓慢
- **文档驱动交流**：通过结构化文档传递信息，效率远高于直接对话

### 核心价值

MemDDC 的核心思想是**提高交流效率，减少大模型和人的低速沟通**。通过以下方式实现：

1. **全量代码预扫描**：首次使用时，自动对项目全量代码进行深度扫描，生成详尽的软件工程文档
2. **智能文档生成**：根据项目类型、编程语言和使用的框架，生成相应的详细文档：
   - Java 后端项目：类说明文档、实体映射文档、方法文档、Spring 框架文档等
   - Python 后端项目：模块说明文档、类和函数文档、Django/Flask 框架文档等
   - 前端项目：组件文档、状态管理文档、API 调用文档、路由文档等
   - 移动应用项目：页面文档、组件文档、API 调用文档、权限文档等
3. **DDD领域模型**：建立清晰的领域边界和业务契约，减少沟通歧义，确保代码修改符合领域设计
4. **智能记忆压缩**：将生成的详尽文档进行智能压缩，保留核心信息，减少冗余内容，生成高效的记忆快照
5. **代码风格统一**：通过自动化工具确保代码风格一致性，减少因风格差异导致的沟通成本

### 工作原理的优势

传统的 AI 编码方式是：
- 从一个点开始，然后通过上下文搜索代码
- 频繁的上下文检索，导致编码速度慢，token 消耗大
- 对项目整体理解不足，容易产生不符合架构约束的代码

MemDDC 的优势：
- **预扫描全量代码**：首次使用时对整个项目进行全量扫描，生成详尽的软件工程文档
- **智能压缩优化**：将详尽文档压缩为核心信息，生成记忆快照，减少存储和传输成本
- **快速记忆加载**：后续编码时直接加载压缩的记忆快照，无需重复扫描代码
- **减少上下文检索**：通过记忆快照提供项目整体上下文，减少频繁的代码检索
- **节省 token 消耗**：压缩后的记忆快照体积小，减少上下文注入的 token 消耗
- **编码又快又准**：AI 能够快速理解项目架构和约束，生成符合设计规范的代码

### 工作原理

1. **初始化阶段**：扫描项目并生成完整文档、DDD模型和记忆快照
2. **迭代阶段**：自动加载现有资料作为上下文约束，确保代码修改符合历史架构决策
3. **闭环阶段**：自动同步更新文档和模型，保持所有资料最新且轻量化

## 如何使用

### 手动触发

通过以下关键词触发 MemDDC Skill：
- `MemDDC`
- `加载记忆约束修改`
- `按DDD契约迭代更新`

### 自动调用设置

#### 编辑器自动调用配置

1. **VS Code 配置**：
   - 安装 TRAE 插件
   - 在 `settings.json` 中添加自动触发配置：

   ```json
   {
     "trae.autoInvokeSkills": [
       {
         "skillName": "MemDDC",
         "triggers": [
           "onFileSave",
           "onCodeEdit",
           "onProjectOpen"
         ],
         "conditions": {
           "fileExtensions": [".js", ".ts", ".jsx", ".tsx", ".java", ".py"],
           "projectTypes": ["node", "react", "angular", "vue", "java", "python"]
         }
       }
     ]
   }
   ```

2. **JetBrains 系列 IDE 配置**：
   - 安装 TRAE 插件
   - 在插件设置中启用自动触发：
     - 触发时机：文件保存、代码编辑、项目打开
     - 适用文件类型：所有源代码文件

3. **Sublime Text 配置**：
   - 安装 TRAE 插件
   - 在插件配置文件中添加：

   ```json
   {
     "auto_invoke": true,
     "skills": [
       {
         "name": "MemDDC",
         "triggers": ["on_save", "on_modify", "on_load"]
       }
     ]
   }
   ```

#### 项目级自动调用

在项目根目录创建 `.trae/config.json` 文件：

```json
{
  "autoInvoke": true,
  "skills": [
    {
      "name": "MemDDC",
      "triggers": [
        "onFileCreate",
        "onFileDelete",
        "onFileRename",
        "onBranchChange",
        "onMerge"
      ],
      "options": {
        "scanDepth": 5,
        "compressionLevel": 7,
        "autoUpdateDocs": true,
        "autoEnforceCodeStyle": true,
        "autoLoadSnapshot": true,
        "loadDocumentTypes": ["architecture", "api", "ddd-model", "class-docs"],
        "cacheSize": 1024,
        "cacheExpiry": 3600,
        "loadStrategy": "snapshot-first"
      }
    }
  ]
}
```

## 技术特点

- **智能文档生成**：自动扫描项目并生成结构化文档

- **复杂架构迭代**：确保架构决策的连续性和一致性

- **并行处理**：同时处理多个文档生成任务
- **智能缓存**：缓存中间结果，提高重复操作效率
- **压缩算法**：使用高效压缩算法，减少存储和传输成本

## 故障排除

### 常见问题

1. **初始化失败**
   - 检查项目目录权限
   - 确保项目中包含足够的源码文件
   - 查看日志文件了解具体错误

2. **记忆加载失败**
   - 检查记忆快照文件是否存在
   - 验证文件格式是否正确
   - 尝试重新初始化项目

3. **文档更新失败**
   - 检查文档目录权限
   - 确保磁盘空间充足
   - 查看日志文件了解具体错误

4. **代码风格检查失败**
   - 检查代码风格配置文件是否正确
   - 确保代码风格检查器有执行权限
   - 查看日志文件了解具体错误

## 应用案例

### 俄罗斯方块游戏开发对比

为了验证 MemDDC Skill 在不同场景下的表现，我们使用两种不同方式开发了相同的俄罗斯方块游戏：

#### 案例一：直接使用 Trae Solo 模式开发

**路径**: `example/teris-game-useTraeSolo/`

**开发方式**:
- 直接使用 Trae IDE 的 Solo 模式，通过自然语言对话生成代码
- 一次性生成完整的游戏代码（HTML + CSS + JavaScript）
- 三个文件结构：index.html、style.css、tetris.js

**特点**:
- 快速原型开发，适合小型项目
- 代码集中在一个文件中，便于快速理解
- 适合个人快速验证想法

#### 案例二：使用 MemDDC Skill 文档驱动开发

**路径**: `example/tetris-game-useMemDDC/`

**开发方式**:
1. **第一阶段 - 文档设计**:
   - 使用 MemDDC Skill 生成完整的 DDD 领域模型 (`ddd-model.md`)
   - 创建架构文档 (`docs/architecture.md`)，定义分层架构
   - 编写 API 文档 (`docs/api.md`)，明确接口契约
   - 生成记忆快照 (`mem-snapshot.json`)，压缩存储项目核心信息

2. **第二阶段 - 代码实现**:
   - 严格按照文档定义的分层架构编写代码
   - 领域层：tetromino.js、board.js（核心业务逻辑）
   - 应用层：game.js（游戏控制器）
   - 基础设施层：renderer.js、input.js（渲染和输入）
   - 表现层：index.html、styles.css

**特点**:
- 清晰的架构设计和职责分离
- 完整的文档体系，便于维护和扩展
- 符合领域驱动设计原则
- 适合团队协作和长期维护

#### 对比总结

| 维度 | Trae Solo 模式 | MemDDC Skill 模式 |
|------|---------------|------------------|
| **开发速度** | 快，一次性生成 | 中等，分阶段进行 |
| **代码结构** | 简单，文件少 | 清晰，分层明确 |
| **架构设计** | 隐含在代码中 | 显式文档化 |
| **可维护性** | 适合小型项目 | 适合中大型项目 |
| **团队协作** | 依赖代码注释 | 有完整文档支持 |
| **扩展性** | 需要重构 | 易于扩展 |
| **适用场景** | 快速原型、个人项目 | 正式项目、团队协作 |

#### 结论

这两个案例展示了 MemDDC Skill 的**双重价值**：

1. **小型项目从 0 到 1**：即使是小型游戏项目，使用 MemDDC Skill 也能产出结构清晰、文档完善的代码，为后续迭代打下良好基础

2. **大型项目重构迭代**：对于复杂项目，MemDDC Skill 的文档驱动方式能够：
   - 建立清晰的领域边界
   - 保持架构决策的一致性
   - 降低团队成员的理解成本
   - 支持长期维护和演进

无论项目规模大小，MemDDC Skill 都能帮助开发者产出更高质量、更易维护的代码。
